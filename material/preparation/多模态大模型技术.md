RLHF 主要包括自监督微调、奖励模型和强化学习三步。自监督微调主要是对预训练数据集进行清洗使其更偏向人类偏好，而奖励模型通过训练最大化正例和负例之间的差距来达到目的，强化学习就是用 Proximal Policy Optimization (PPO)算法最优化强化学习策略模型

DPO 直接从人类偏好标签中学习一个二分类任务，不需要考虑强化学习中的奖励模型和策略模型，使其能够保持简单在数据集的选择上，对于现有数据集进行清洗处理使其满足要求比较贵，因此可选择面上比较狭窄2.4 总结

def pixel_shuffle1(x, scale_factor=2):
    bsz, seq, embed_dim = x.size()
    height = width = int(seq**0.5)
    x = x.view(bsz, height, width, embed_dim).contiguous()
    x = x.view(bsz, height, int(width / scale_factor), embed_dim * scale_factor)
    
    x = x.permute(0, 2, 1, 3).contiguous()
    x = x.reshape(bsz, int(width / scale_factor), int(height / scale_factor), embed_dim * (scale_factor**2))
    x = x.permute(0, 2, 1, 3).contiguous()

    return x

4.3.1 Token 压缩 vs KV Cache压缩
- Vision Token 压缩（例如，FastV，VisionZip，PruMerge +）
  - 结论1: Token压缩性能与任务相关，并且对基准和模型表现出显著的敏感性。大多数方法在较高预算下表现稳定，但在极低预算（例如 1%）下性能会急剧下降，尤其是在需要细粒度视觉细节或长输出的任务上。对于 1% 保留下的标记剪枝，视觉编码器中的剪枝方法（例如 VisionZip、PruMerge+）优于 LLM 主干中的剪枝方法（例如 FastV）。
- KV Cache 压缩（例如，StreamingLLM，H2O，SnapKV，PyramidKV，LOOK-M，VL-Cache）。
  - 结论2: 在泛化能力和准确性方面，KV Cache压缩优于 token 压缩。H2O 和 PyramidKV 等方法表现出色。当泛化能力和忠诚度至关重要时，KV Cache压缩方法通常是首选。
- 结论3: 根据实际任务情况选择Token 压缩 或 KV Cache 压缩，可以实现更佳的性能与效率平衡。Token 压缩可以显著缩短首次令牌生成时间 (TTFT)，非常适合短响应任务。KV Cache 压缩方法则可能在低预算且输出较长的任务中提供更佳性能。

MM-DPO是对传统DPO框架的扩展，重点引入动态奖励缩放（Dynamic Reward Scaling），该方法根据各比较对的奖励信号动态调整样本的损失权重。传统DPO方法对所有训练对采用固定的训练权重，而MM-DPO在训练过程中优先考虑那些具有更大奖励边际的比较对，提升了模型更新的有效性。
核心-MM-DPO：有效利用高质量偏好数据
[Image]
要有效利用MM-RLHF中的高质量数据，我们有以下的实验发现和技巧
1. MM-DPO不再仅仅关注“最难的比较对”（即排名差异最大的一对），而是将一个查询下所有可能的响应对都纳入训练。具体来说，对于一个查询 x，如果有多个响应，每一对具有不同排名的响应都被视为一个有效的比较对。这种全面的处理方式可以捕捉更细粒度的排序信息，让模型从更广泛的偏好数据中学习。然而，这种策略也带来了新的挑战：当响应对的排名差异较小时（例如排名 3 和排名 4 的比较），其奖励差距（reward margin）往往较小，而排名差异较大的响应对（例如排名 1 和排名 4 的比较）包含的信息质量更高。如果对所有样本对一视同仁，会导致高置信度的信息被低效利用。
2. 为了解决这个问题，MM-DPO 引入了动态奖励缩放（Dynamic Reward Scaling）机制，根据奖励差距动态调整更新强度，优先利用高置信度的样本对。具体而言，奖励模型可以自然地为样本对提供奖励差距（reward margin），这为动态控制样本的更新权重提供了一个直接的信号。

4.6 架构探索
[Image]
LVLMs各种视觉整合技术的比较。
( a )架构注入：在LLM中插入额外的层以实现跨模态交互；
( b )上下文注入：将视觉令牌串接在文本序列之前作为初始上下文。（现在的主流架构）
( c )特征调制注入：通过视觉引导的仿射变换对内部隐藏状态进行调制。
效果：Incontex-Injection > Feature-Injection > Architectural injection
性能：Feature-Injection >Architectural injection~=Incontex-Injection 
Architectural injection
早期的CrossAttention，在transformer block中串行引入计算模块，可能会破坏原始LLM架构连贯性。
新的架构注入：LoRA / MOE 架构，并行注入计算，能兼顾最小结构 干扰。
但是效果上限不如 Incontext-Injection方式。

Incontex-Injection
目前主流的做法，效果最好，性能受到Vision token长度影响，特别是在高分辨率图像或长视频序下，会导致的推理延迟和计算瓶颈。

Feature-Injection
参数注入的方式，避免了Vision token引入的计算开销，也能兼顾最小结构 干扰 & 计算高效性。
效果基本 能对齐Incontext-Injection 的早期工作 LLaVa 1.6
