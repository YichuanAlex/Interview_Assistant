特征数据（Feature Data）：主要由用户的行为记录（如曝光、点击、停留时长等）以及在此基础上构造的各种派生特征组成，通常以 TBase 宽表的形式存储，是推荐系统的核心输入。

KV Cache 数据（KV Cache）：是 Transformer 架构特有的一类数据，用于在多层自注意力计算中缓存中间结果，典型做法是用存储换计算，提前持久化这些中间特征，在后续推理请求中复用，从而显著减少重复计算开销。

LRM 场景下，数据库面临三大核心挑战：

混合访问模式冲突：训练侧需要按时间批量扫描全量用户、推理侧又要求按用户低延迟点查，传统行存/列存难以同时兼顾；

规模与成本压力：上百个模型叠加形成百 PB～EB 级特征与 KV Cache，只能主要依赖低成本 HDD，却要在随机读、长尾延迟和写带宽“三座大山”下仍提供可接受的性能；

资源治理难题：在线推理、离线训练和后台任务在同一集群上争抢 CPU、缓存和 I/O，若缺乏精细化的优先级与限流机制，高优先级服务的吞吐会被拖垮、尾延迟会失控。

围绕上述三大挑战，我们在原有的 TokaDB 的基础上，设计了三个模块，分别解决三大挑战：

混合访问专用的数据组织 (Hybrid-Access Specialized Data Organization): 针对用户行为序列的“行存 vs. 列存”两难问题，从数据布局层面进行创新。

成本-性能协同优化的 I/O 引擎 (Cost-Efficient Cooperative I/O Engine): 针对 EB 级 KV Cache 在 HDD 上的成本与性能矛盾，从 I/O 路径上进行极致优化。

基于优先级的资源管理器 (Priority-Based Resource Manager): 针对高优先级任务的性能保障难题，对 CPU、缓存、I/O 三大资源进行精细化的优先级管控。

2.2.1 数据组织：提升混合访问性能2.2.1.1 混合访问模式冲突[Image][Image]在 LRM 的用户行为序列上同时存在两种主导访问模式：点查（Point Read） 与 范围扫描（Range Scan）。这两类访问对底层数据组织的偏好天然冲突：

推理侧常见的是“读取单个用户的完整行为序列”以捕捉最新兴趣，倾向于行式组织，适合快速按用户 ID 定位聚合（蓝色 point read）。

训练侧常见的是“批量扫描所有用户在某一时间段的行为”以构建样本与特征拼接，倾向于列式/时间分段组织（绿色 scan read）。现有开源系统在数据组织上的取舍导致两端难以兼顾：

行存（如 HBase）按行键将同一用户的记录聚合在一起，Point Read 快；但在 Range Scan 时会连带读取大量与当前训练目标无关的列/记录，造成带宽浪费。

列存（如 Doris）按列/分区（常以时间或块为单位）聚合，Scan 高效；但 Point Read 时需要将同一用户分散在多个分区/列组的记录重新拼接聚合，延迟显著上升。

一些行列混存的方案通过双份冗余同时保留行式与列式布局，带来翻倍空间成本与写入复杂度，在大规模 LRM 数据上难以长期承受。

2.2.2.2 混合访问专用的数据组织[Image]为了同时提升点查和范围扫描的性能，TokaDB 设计了 Detached-Segment Index Layout（索引与数据分离布局）。

核心思想:

聚合与分段: 首先，我们将一个用户的所有行为记录在逻辑上聚合存储，保证了点查所需的数据局部性。然后，我们将这些聚合后的记录按时间切分成多个数据段 (Segment)。

索引与数据分离: 我们为每个数据段提取元信息（如时间范围、存储地址），形成一个轻量的“段索引”，并将其存储在 LSM-Tree 中。而真正庞大的数据段本身，则顺序存储在专用的 ValueLog 文件里。

如何解决问题:

点查时: 只需通过 uid 在 LSM-Tree 中快速找到轻量级的段索引，然后根据索引直接读取所需的数据段即可，这些数据段在物理上也是相邻的。这个过程保持了行存的局部性优势，避免了列存的 Join 开销。

范围扫描时: 只需遍历轻量级的索引层，根据时间范围快速过滤掉大量无关的数据段，只读取真正需要的段。这个过程避免了行存的读放大问题，实现了类似列存的高效过滤。此外，我们还设计了局部性触发的垃圾回收 (LTGC) 机制，主动整理因更新而变得分散的数据段，持续保持点查性能；同时通过 SSD/HDD 冷热分层，将近期热数据放 SSD、远期冷数据放 HDD，进一步平衡成本与性能。

2.2.2 IO 引擎：性能和成本协同优化2.2.2.1 规模与成本压力[Image]字节内部存在上百个 LRM 模型并行训练与滚动上线，且每个模型在推理侧维护独立的 KV Cache。这使得KV Cache 总规模随模型数与版本迭代呈叠加增长，形成超大体量的持久化中间特征，可达百 PB 到 EB 级别；

SSD 成本过高: 我们做过详细的成本评估，如果将 EB 级的 KV Cache 全部存储在 SSD 上，其硬件成本几乎相当于一个配备了 NVIDIA A100 的 GPU 集群价格的 70%。这在经济上是完全不可行的。

HDD 性能瓶颈: 因此，我们只能选择成本低廉的 HDD。但 HDD 的物理特性带来了“三座大山”：

随机读性能差: HDD 的 IOPS 极低，无法支撑 KV Cache 所需的高并发随机读取。

长尾延迟不可控: 磁盘“慢节点”（如瞬时故障、后台任务干扰）现象频发，导致 P99 延迟急剧恶化，严重影响在线服务质量。

写入带宽不足: 读请求已经消耗了大部分带宽，而传统数据库为了保证数据一致性，一次写入需要持久化两次（写日志 + 写数据文件），这种双写开销进一步榨干了本就稀缺的写入带宽。

2.2.2.2 协同优化的 I/O 引擎在基于廉价 HDD 的前提下实现高性能，需要从读写 I/O 的全路径进行协同优化。挑战技术方案思路随机读性能差硬件感知的块对齐 (Hardware-Aware Block Alignment)我们强制保证每个 KV Cache 记录都完整地存储在单个物理块内，从根源上消除了因跨块存储而导致的多次磁盘寻道，确保每次读取都是一次最高效的单次 I/O长尾延迟不可控自适应备份读 (Adaptive Backup Reads)当一个读请求超时未返回时，立即向另一个副本发出备份请求。关键在于，这个超时阈值 (deadline) 会根据当前系统负载自适应调整，避免在系统繁忙时触发过多备份请求，引发“雪崩效应”。写入带宽不足日志即数据 (Log as Data):我们抛弃了传统数据库“写日志 + 写数据文件”的双写模式。我们将日志文件本身就作为主数据，LSM-Tree 中只存储指向日志中数据位置的轻量级指针。这样一来，每次写入只需一次持久化，写带宽理论上直接翻倍。[Image]2.2.3 资源调度：在离线一体化2.2.3.1 资源治理难题在 LRM 中，不同工作负载的优先级并不一致。Serving/在线推理优先追求低延迟与稳定性；训练/离线任务侧重吞吐与带宽，对时延更容忍。在现有系统中，高优先级的服务请求，需要和低优先级的训练任务、后台 Compaction/GC 任务，共同争抢 CPU、缓存和 I/O 资源。我们观测到的典型问题包括：

CPU 争抢: 缺乏有效的抢占式调度，高优先级的服务请求会被低优先级任务长时间阻塞。

缓存污染: 共享的 Block Cache 极易被低优先级任务的大规模读写“污染”，导致高优先级任务的热数据被频繁换出，缓存命中率断崖式下跌。

I/O 争抢: 简单的速率限制无法区分请求的“贵贱”，低优先级的流量洪峰一来，会直接挤占高优先级任务的存储带宽通道。[Image]实验数据显示，在资源争抢的情况下，线上服务的点查吞吐量会下降 60%，P99 延迟会飙升 120%。这种严重的性能抖动，在生产环境中是绝对无法容忍的。

2.2.3.2 基于优先级的资源管理器[Image]为高优先级的在线服务提供可靠的性能保障，TokaDB 对 CPU、缓存和 I/O 三大核心资源进行了全面的隔离设计。

CPU 隔离：抢占式线程调度器: 我们实现了一个用户态的抢占式调度器。当高优先级的服务请求到达时，可以立即抢占正在运行的低优先级训练或后台任务，无需等待其执行完毕，确保服务请求能第一时间得到 CPU 响应。

缓存隔离：分层缓存分配器: 我们将缓存划分为三个独立的池：

元数据池 (Metadata Pool): 最高优先级，存放索引等核心元数据，常驻内存 (Pinned)，确保查找路径最快。

服务数据池 (Serving Data Pool): 占比最大，专门服务于在线请求，采用 LRU 策略。

分段池 (Staging Pool): 占比最小，供训练和后台任务使用，采用 FIFO 策略。它即使被写满也绝不会污染到服务数据池，从而杜绝了“缓存污染”问题。

I/O 隔离：多层集成速率限制器: 我们同时在请求层和硬件层进行限流。硬件层的限流器会实时监测磁盘的负载。当发现磁盘快要被打满时，它会通过反向施压 (Backpressure)机制，主动通知并降低低优先级请求队列的令牌发放速率，从而为高优先级任务留出畅通的 I/O 通道。
